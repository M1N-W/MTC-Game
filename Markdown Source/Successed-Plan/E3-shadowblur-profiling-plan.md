# E3 — shadowBlur Audit & Reduction (Profiling-Backed PR Plan)

*Groundwork draft prepared alongside v3.44.1 ship — NOT yet implemented.*

## Why this is a separate PR

`shadowBlur` writes have a visual surface of **455 sites across 5 files**.
Several already carry hand-tuned `// PERF: 20→10` comments from earlier
passes, so a naive bulk cut will regress visuals without measurable gain.
The v3.44.1 plan deferred this work explicitly: *"E items should ship with
before/after INP numbers."*

## Profile collection protocol (to run on target machine)

1. Start local server: `python -m http.server 8765`
2. Open `http://localhost:8765/` in Chrome → DevTools → **Performance** tab.
3. Select the **Auto** character (highest shadowBlur surface).
4. `sudo next` in admin terminal to skip to wave 4–5 (mid-density, no boss).
5. Click **Record** → let run for **10 s** → Stop.
6. Save trace as `v3.44.1-baseline-wave4.json`.
7. Repeat for: `Pat wave 6 (boss)`, `Kao wave 10 (glitch)`, `Poom wave 3 (boss)`.

## Key metrics to extract

From the Performance trace summary:

- **Scripting time** in `updateGame` frame block
- **Rendering / Painting time** total per frame
- Bottom-up view: sort by **Self Time** → look for `CanvasRenderingContext2D.set shadowBlur` and `2D.fillRect` / `2D.stroke` sibling costs.
- Screenshot timeline — confirm visual parity.

## Proposed targeted cuts (ranked by effort × impact)

### Priority 1 — EnemyRenderer (×N multiplier)

| Function | Site range | Proposed change | Risk |
|----------|-----------|-----------------|------|
| `drawEnemy` (L.370–L.420 area) | 10 sites | Skip outer-glow & charge arc `shadowBlur` when LOD-far flag is set (already computed) — expand v3.44.0 LOD Tier | Low |
| `drawTank` | 12 sites | Same LOD gate; also cache tank-rivet `shadowBlur=0` reset pattern | Low |
| `drawMage` | 13 sites | Same LOD gate; rotation glow is most expensive | Med (visual) |
| `_drawStatusOverlays` | 6 sites | Reset to 0 only once at end of loop, not per-overlay | Low |

**Expected win:** ~30–40 ops × 40 enemies = **1200–1600 ops/frame saved** in dense waves.

### Priority 2 — BossRenderer (×1, but large)

| Function | Site range | Proposed change | Risk |
|----------|-----------|-----------------|------|
| `drawBossFirst` | 59 sites | Group sequential `shadowBlur = X; shadowColor = Y;` pairs — if same color, only write when changed | Med |
| `drawBoss` | 38 sites | Same dedupe pass | Med |

Boss runs only 1×/frame, so total savings is capped. Still worth it because the boss is usually on-screen and dominates per-frame cost during boss waves.

### Priority 3 — PlayerRenderer

All player-render shadowBlur runs 1×/frame, so **not worth risking visual
regression** unless profile shows paint-time hot spot. Suggest **keep as-is**
unless E1/E2 + Enemy cuts are insufficient.

### Priority 4 — effects.js & ui.js

effects.js `draw` loops over particles — 34 sites but most are inside
`if (p.glow)` branches. Check whether particle pool actually enables glow
for majority (probably no) → likely already cheap. **Keep as-is**.

ui.js `_minimapDrawContent` already covered by E2 throttle (now runs at
~15 Hz not 60 Hz) — **no additional work needed**.

## Acceptance criteria for the E3 PR

1. Scripting-time reduction ≥ 10 % on the `v3.44.1-baseline-wave4` trace, measured on the same machine with the same setup.
2. No visual regression in a side-by-side screenshot diff (use Chrome DevTools **Rendering → Paint flashing** + screenshot compare).
3. Smoke test (`.tmp_smoke.py`) still 13/13 PASS.
4. All `// PERF:` comments updated with new numbers when a site is touched.

## Rollback plan

Each targeted cut should be a separate git commit inside the PR. If a visual regression is flagged in review:

```
git revert <commit-hash>   # keeps the others
```

## Non-goals (explicit)

- Adding a graphics-quality slider in settings (separate feature, not perf).
- Migrating to `OffscreenCanvas` workers (much larger refactor).
- Replacing `shadowBlur` with pre-rendered sprite glows (also separate).
